## 큰 그림: 4단계 로드맵

### Phase 4: 기능 완성
> 목표: 매일 쓰는 핵심 기능 빈틈 채우기. 프론트엔드 위주로 빠르게 완성

- [ ] **4-1. 발송 관리**
  - shipments 테이블 생성
  - 거래별 발송 등록 (택배사, 송장번호)
  - 부분발송 지원 (10개 주문 중 3개만 먼저 발송)
  - 거래 목록에 발송 상태 표시 (대기/부분발송/완료)
  - 발송 완료 시 안내 메시지 자동 생성 (카톡 복사용)

- [ ] **4-2. 가격 변동 이력** (plan-price-history.md 기반)
  - price_history 테이블 생성
  - 가격 변경 시 이력 기록 (프론트엔드에서 저장 시 함께 기록)
  - 상품 상세 페이지에 이력 표시
  - 거래처 페이지에 "가격 변동 안내" 표시 (공개 설정된 것만)

- [ ] **4-3. 별칭 관리 UI**
  - 상품 상세에서 별칭 목록 조회/추가/삭제
  - 전체 별칭 검색 페이지 (어떤 별칭이 어떤 상품에 연결되어 있는지)

### Phase 5: 업무 효율 향상
> 목표: 반복 작업을 줄이고, 한눈에 파악할 수 있게

- [ ] **5-1. 대시보드 고도화**
  - 오늘/이번 주/이번 달 매출
  - 마진 분석 (총 마진, 마진율)
  - 미수금 TOP 거래처
  - 발송 대기 목록
  - 베스트셀러 / 판매 부진 상품

- [ ] **5-2. 거래처 통계**
  - 거래처별 누적 거래액, 거래 건수
  - TOP 5 구매 상품
  - 월별 거래 추이 그래프

- [ ] **5-3. 엑셀 일괄 업로드**
  - 상품 + 가격 일괄 등록/수정
  - 거래처별 시트에서 특별단가 일괄 설정

- [ ] **5-4. 명세서 이메일 발송**
  - 명세서를 이메일로 직접 보내기
  - 발송 기록 저장

### Phase 6: 백엔드 구축 + 권한/보안
> 목표: Supabase 백엔드 통일. 기능이 안정된 후 서버 측 보호 계층 추가
> 시점: 기능 완성 후, 직원 투입 전

- [ ] **6-1. DB 트리거 / RPC 적용**
  - 거래 보호 트리거 실행 (SQL 이미 준비됨: `20260407_protect_invoiced_transactions.sql`)
  - 가격 변경 이력 자동 기록을 DB 트리거로 전환
  - 발송 수량 초과 방지 트리거
  - 비즈니스 규칙을 DB 수준으로 이동 (프론트엔드 검증 → DB 검증)

- [ ] **6-2. 사용자 권한 시스템**
  - users 테이블 + Supabase Auth 연동
  - RLS 정책으로 역할별 접근 제어 (DB 수준)
  - 관리자: 모든 기능
  - 직원: 조회 + 거래입력 + 발송등록만

- [ ] **6-3. Edge Functions 이전**
  - AI 주문 파싱: Next.js API Route → Supabase Edge Function
  - 이메일 발송: Edge Function으로 구현

- [ ] **6-4. 데이터 보호**
  - 삭제 시 soft delete (is_active 플래그)
  - 중요 변경사항 로그 기록

### Phase 7: 모바일 최적화 및 마무리
> 목표: 핸드폰에서 주로 사용하므로 모바일 경험 완성

- [ ] **7-1. 모바일 UI 최적화**
  - 터치 영역 확대
  - 핵심 플로우 모바일 테스트 (주문인식 → 거래저장 → 명세서)
  - 하단 고정 내비게이션 검토

- [ ] **7-2. 성능 최적화**
  - 데이터 로딩 속도 개선
  - 이미지/번들 최적화

- [ ] **7-3. 에러 처리 강화**
  - 네트워크 실패 시 재시도
  - 사용자 친화적 에러 메시지

```
    │

    │

    │
Phase 6 (백엔드+보안) ─ DB 트리거, 권한, Edge Functions, soft delete
    │
Phase 7 (마무리) ────── 모바일, 성능, 에러
```

## 우선순위 판단 기준

1. **매일 쓰는 기능인가?** → 최우선 (발송관리)
2. **돈과 직접 관련 있는가?** → 높음 (가격이력)
3. **현재 수작업으로 하고 있는가?** → 중간 (엑셀, 이메일)
4. **있으면 좋지만 없어도 되는가?** → 낮음 (통계, 그래프)
5. **기능 안정화 후 해도 되는가?** → 나중 (백엔드 구축, 권한)
