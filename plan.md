## 현재 상태 요약

**완성된 기능 (Phase 1~3 완료)**
- 로그인/인증 (Supabase Auth)
- 거래처 CRUD + 가격등급 + 특별단가
- 상품 CRUD + 옵션 + 옵션별 가격
- 거래 CRUD (수동입력 + AI 주문인식)
- 명세서 생성/상세 (PNG/PDF 다운로드, 미수금 표시)
- 입금 관리
- 미수금 현황
- 상품 별칭 (AI 매칭용, DB에만 존재)
- 네비게이션 (PC + 모바일 반응형)
- 회사 정보 설정
- 거래 수정/삭제 보호 (명세서 연결 시 UI 잠금 + DB 트리거 SQL 준비됨)
- API 인증 추가 (parse-order)
- AI 빈 응답 에러 처리
- 모바일 메뉴 자동 닫기

## 큰 그림: 4단계 로드맵

### 즉시 처리
> 작업량 적고 체감 효과 큰 항목

- [ ] **명세서 시작일 자동 설정** — 거래처 선택 시 명세서 미연결 거래 중 가장 오래된 날짜를 시작일 기본값으로

### Phase 4: 기능 완성
> 목표: 매일 쓰는 핵심 기능 빈틈 채우기. 프론트엔드 위주로 빠르게 완성

- [ ] **4-1. 별칭 관리 UI**
  - 상품 상세에서 별칭 목록 조회/추가/삭제
  - 전체 별칭 검색 페이지 (어떤 별칭이 어떤 상품에 연결되어 있는지)

- [ ] **4-2. 발송 관리**
  - shipments 테이블 생성
  - 거래별 발송 등록 (택배사, 송장번호)
  - 부분발송 지원 (10개 주문 중 3개만 먼저 발송)
  - 거래 목록에 발송 상태 표시 (대기/부분발송/완료)
  - 발송 완료 시 안내 메시지 자동 생성 (카톡 복사용)

- [ ] **4-3. 가격 변동 이력 + 명세서/거래 안정성** (plan-price-history.md 기반)
  - price_history 테이블 생성
  - 가격 변경 시 이력 기록 (프론트엔드에서 저장 시 함께 기록)
  - 상품 상세 페이지에 이력 표시
  - 거래처 페이지에 "가격 변동 안내" 표시 (공개 설정된 것만)
  - [ ] 명세서 생성 시 데이터 꼬임 방지 (DB 트랜잭션/RPC로 명세서+거래 연결을 원자적 처리)
  - [ ] AI 주문 일괄 저장 중간 실패 대책 (일괄 insert 또는 롤백 로직 추가)

### Phase 5: 업무 효율 향상
> 목표: 반복 작업을 줄이고, 한눈에 파악할 수 있게

- [ ] **5-1. 대시보드 고도화**
  - 오늘/이번 주/이번 달 매출
  - 마진 분석 (총 마진, 마진율)
  - 미수금 TOP 거래처
  - 발송 대기 목록
  - 베스트셀러 / 판매 부진 상품

- [ ] **5-2. 거래처 통계**
  - 거래처별 누적 거래액, 거래 건수
  - TOP 5 구매 상품
  - 월별 거래 추이 그래프

- [ ] **5-3. 엑셀 일괄 업로드**
  - 상품 + 가격 일괄 등록/수정
  - 거래처별 시트에서 특별단가 일괄 설정

- [ ] **5-4. 명세서 이메일 발송**
  - 명세서를 이메일로 직접 보내기
  - 발송 기록 저장

- [ ] **5-5. 미수금 계산 성능 개선**
  - 클라이언트에서 전체 데이터 불러와 계산하는 방식 → DB에서 GROUP BY로 집계
  - 거래처/명세서/입금이 늘어나도 빠르게 동작하도록

- [ ] **5-6. alert() → 토스트 메시지로 전환**
  - 저장 성공/실패, 삭제 등 모든 alert()을 토스트 메시지로 교체

- [ ] **5-7. lookupPrice를 하나의 RPC로 통합**
  - 현재 DB 5~7번 호출을 1번으로 줄임 (Supabase RPC 함수로 가격 조회 로직 통합)

### Phase 6A: 데이터 보호 강화
> 목표: 데이터 안전장치 확보
> 시점: 기능 완성 후, 직원 투입 전

- [ ] **DB 트리거 실행**
  - 거래 보호 트리거 (SQL 이미 준비됨: `20260407_protect_invoiced_transactions.sql`)
  - 발송 수량 초과 방지 트리거
- [ ] **soft delete 적용** (is_active 플래그)
- [ ] **RLS 기본 설정** (인증된 사용자만 접근)

### Phase 6B: 권한 시스템
> 목표: 역할별 접근 제어
> 시점: 직원 투입 시

- [ ] **사용자 권한** (관리자/직원 역할)
- [ ] **RLS 역할별 정책 세분화**
  - 관리자: 모든 기능
  - 직원: 조회 + 거래입력 + 발송등록만

### Phase 6C: 코드 구조 정리
> 목표: 기술 부채 해소
> 시점: 여유 있을 때

- [ ] **AI 주문 파싱**: Next.js API Route → Supabase Edge Function 이전
- [ ] **이메일 발송**: Edge Function으로 구현
- [ ] **서버 컴포넌트 전환 검토**

### Phase 7: 모바일 최적화 및 마무리
> 목표: 핸드폰에서 주로 사용하므로 모바일 경험 완성

- [ ] **7-1. 모바일 UI 최적화**
  - 터치 영역 확대
  - 핵심 플로우 모바일 테스트 (주문인식 → 거래저장 → 명세서)
  - 하단 고정 내비게이션 검토

- [ ] **7-2. 성능 최적화**
  - 데이터 로딩 속도 개선
  - 이미지/번들 최적화

- [ ] **7-3. 에러 처리 강화**
  - 네트워크 실패 시 재시도
  - 사용자 친화적 에러 메시지

- [ ] **7-4. 코드 리뷰 지적사항 일괄 정리**
  - [ ] DB 에러 무시 수정: 전 페이지에서 Supabase 쿼리 시 `error` 체크 + 사용자 안내 추가
  - [ ] 가격 조회 경쟁 상태: 빠르게 선택 변경 시 이전 응답이 덮어쓰는 문제 방지
  - [ ] useParams() id 배열 방어: `string | string[]` 타입 가드 추가
  - [ ] 명세서 기간 검증: 시작일 > 종료일이면 경고
  - [ ] 입금액 음수 검증: 0 이하 금액 입력 방지
  - [ ] 공통 유틸/타입 분리: `formatPhone`, `formatPrice`, `rawPrice`, `getProductName` 등 중복 코드 제거
  - [ ] 가격 등급 하드코딩 제거: `level === 0`, `level === 1` 상수화 또는 DB 조회로 통일
  - [ ] 상품 삭제 시 거래 기록이 있으면 삭제 차단, 없으면 관련 데이터 연쇄 삭제

## 진행 순서

```
즉시처리 (명세서시작일) ── 1줄 수정급, 바로 체감
    │
Phase 4-1 (별칭관리) ──── 작업량 적음, AI 인식률 향상
    │
Phase 4-2 (발송관리) ──── 매일 쓰는 핵심 기능
    │
Phase 4-3 (가격이력) ──── 돈 관련, 상세 계획 준비됨
    │
Phase 5 (업무효율) ────── 대시보드, 통계, 엑셀, 이메일, 토스트, RPC
    │
Phase 6A (데이터보호) ─── DB 트리거, soft delete, RLS 기본
    │
Phase 6B (권한) ────────── 직원 투입 시 역할별 접근 제어
    │
Phase 6C (코드정리) ───── Edge Functions, 서버 컴포넌트 전환
    │
Phase 7 (마무리) ──────── 모바일, 성능, 에러, 코드 리뷰 정리
```

## 우선순위 판단 기준

1. **매일 쓰는 기능인가?** → 최우선 (발송관리)
2. **돈과 직접 관련 있는가?** → 높음 (가격이력)
3. **현재 수작업으로 하고 있는가?** → 중간 (엑셀, 이메일)
4. **있으면 좋지만 없어도 되는가?** → 낮음 (통계, 그래프)
5. **기능 안정화 후 해도 되는가?** → 나중 (백엔드 구축, 권한)
